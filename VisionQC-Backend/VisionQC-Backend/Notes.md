## JWT: 

A token is a code that tells which user is this, his role and where the token expires (session time)
It follows this format: header.payload.signature
Its like document (header and payload) + stamp (signature)

1. Header :Says what type of token and signing algorithm. Example: { "alg": "HS256", "typ": "JWT"}
2. Payload : Contains claims/data, like: { "sub": "user-id", "email": "user@example.com", "role": "Regular", "exp": 123456789 }
3. Signature : The signature is generated by HMAC SHA256 using the encoded header, encoded payload, and secret key (its a string that does not resemble them  (2+2+2=6))

If someone changes the payload, the backend recalculates the signature and sees:

expected signature != token signature
Then rejects the token



So the backend checks the token of every user if validate (so logged in before + in the time of the session-> login), if not log in again
The backend here is builder.Services.AddAuthentication() that checks the jwt , and this functions is called by app.UseAuthentication()

The token is created by an issuer which is here in AuthController.cs, GenerateJwtToken(User user) method
the issuer have to open use key (like a password) to actually validate the token
this is done to block anyone with no access to create a token

## Forgot password flow : 

1. User clicks “Forgot password”.
2. Frontend asks backend to reset password.
3. Backend creates a reset token.
4. Backend builds a reset link using `Frontend:ResetPasswordUrl` (visionqcmobile://reset-password?token=abc123)
5. Backend sends that link by email.
6. User clicks the link.
7. Browser opens the reset password page.

## Web push:

Web push is the notification - reminder system

1. Push notification 

Notification = the popup/message
Push notification = a notification sent automatically from backend to browser/device without the user opening the app and asking for it

2. Public/Private key

Because chrome sends the notification , google needs some identifier 
The mail (Subject) here is the identifier
When sending a notification, the backend uses the `PrivateKey` and `Subject` to sign the notification request.

The VAPID public key identifies your backend as the push sender to Google’s push service.
The private Key is used to check our backend : Does the private key signature match the public key used for this subscription?

So it proves: The sender now = the same sender identity used during subscription



3. p256dh/auth keys

P256dh = public key for this browser subscription
Auth = small browser-generated secret used to strengthen encryption

Google is only the delivery middleman. It should not read the message content. So the backend encrypts the notification in a way that only the target browser can decrypt.

Only the target browser/device has the matching private/internal key material to decrypt it
